# AISDLC-218: Workflow Changes Required for Draft-PR Flow

Generated by the AISDLC-218 implementation. This file documents the edits each
`.github/workflows/*.yml` file needs to skip draft PRs (so CI fires exactly
once per PR — on the `ready_for_review` transition, not on `opened`).

> **IMPORTANT:** The agent that produced this file CANNOT edit `.github/workflows/**`
> (PreToolUse hook blocks it). The operator must apply these edits manually, or
> dispatch a follow-up task that carries the changes.

## Canonical pattern

Use the trigger-level pattern for maximum efficiency — the workflow startup
overhead is eliminated entirely for draft events, not just the job cost:

```yaml
on:
  pull_request:
    # Add ready_for_review to the types list so the workflow fires when the
    # draft is flipped to ready. Keep synchronize + reopened so re-pushes to
    # a ready PR still trigger. Drop `opened` — it would fire on draft opens.
    types: [synchronize, reopened, ready_for_review]
    # ... (preserve existing branches:, paths:, paths-ignore: keys)

jobs:
  <job-name>:
    # Guard at the job level too — covers the merge_group event and any
    # future trigger added by a sibling PR.
    if: github.event.pull_request.draft == false
    # ... rest of job
```

## Audit results: 8 workflows

### 1. `.github/workflows/ai-sdlc-review.yml`

**Current trigger:**
```yaml
on:
  pull_request:
    types: [opened, synchronize, reopened]
    branches: [main]
    paths-ignore:
      - 'spec/rfcs/**'
      - ...
  merge_group:
    types: [checks_requested]
```

**Current draft handling:** None. The workflow fires on `opened` events
including draft PR opens. No job-level `if: !draft` guard.

**Recommended change:**
1. Replace `types: [opened, synchronize, reopened]` with
   `types: [synchronize, reopened, ready_for_review]`.
2. Add `if: github.event.pull_request.draft == false` to the `docs-only-check`
   job AND every other job in the file (note: jobs already gated on
   `github.event_name == 'pull_request'` can fold the draft guard into an `&&`
   expression).

**Rationale:** `ai-sdlc-review.yml` spawns 3 reviewer subagents. Running them
on a draft open (before attestation signs) wastes reviewer tokens and triggers
the `Post Review Results` status check posting before the signing envelope
exists at HEAD. Once AISDLC-218 is fully live, `opened` is never a draft-open;
removing it is a clean no-op for ready PRs and eliminates draft noise entirely.

### 2. `.github/workflows/verify-attestation.yml`

**Current trigger:**
```yaml
on:
  pull_request:
    types: [opened, synchronize, reopened]
    branches: [main]
  merge_group:
    types: [checks_requested]
```

**Current draft handling:** None.

**Recommended change:**
1. Replace `types: [opened, synchronize, reopened]` with
   `types: [synchronize, reopened, ready_for_review]`.
2. Add `if: github.event.pull_request.draft == false` to the `verify` job
   (alongside the existing release-please skip predicate — combine with `&&`).

**Rationale:** `verify-attestation.yml` posts the `ai-sdlc/attestation` required
check. If it fires on draft open (before the envelope chore commit lands), it
posts `failure` — this is CI run #1. After the envelope push, it fires again and
posts `success` — CI run #2. Removing `opened` + adding `ready_for_review` means
it only fires once: after `gh pr ready`, by which time the envelope is already at
HEAD and the check passes immediately.

**Note on `merge_group`:** The `merge_group` event is unaffected by PR draft
state (the PR is always ready before entering the merge queue). No change needed
for the `merge_group` trigger.

### 4. `.github/workflows/ci.yml`

**Current trigger:**
```yaml
on:
  push:
    branches: [main]
  pull_request:
    branches: [main]
  merge_group:
```

**Current draft handling:** None. Note: the `pull_request` trigger here uses no
`types:` key — it defaults to `[opened, synchronize, reopened]`.

**Recommended change:**
1. Add explicit `types: [synchronize, reopened, ready_for_review]` under the
   `pull_request:` key.
2. Add `if: github.event.pull_request.draft == false || github.event_name != 'pull_request'`
   to each job (the `|| github.event_name != 'pull_request'` guard ensures the
   job still runs on `push:` and `merge_group:` events, which don't have a
   `pull_request.draft` context).

**Alternative:** Add a top-level `if:` on each job:
```yaml
if: >-
  github.event_name != 'pull_request' ||
  github.event.pull_request.draft == false
```

**Rationale:** `ci.yml` runs build + test + lint on every PR push. Running on
draft PR opens (before attestation) wastes CI minutes and produces a spurious
`failed` status. The `push: branches: [main]` trigger is unaffected by PR draft
state.

### 7. `.github/workflows/auto-enable-auto-merge.yml`

**Current trigger:**
```yaml
on:
  pull_request:
    types: [opened, synchronize, reopened]
```

**Current draft handling:**
```yaml
if: >-
  github.event.pull_request.draft == false &&
  github.event.pull_request.head.repo.full_name == github.repository
```

Job-level draft guard is ALREADY PRESENT (`draft == false`). However, the
trigger does not include `ready_for_review`, which means when a draft PR is
flipped to ready, this workflow does NOT fire — auto-merge is never enabled.

**Recommended change:**
1. Add `ready_for_review` to the `types:` list:
   ```yaml
   types: [opened, synchronize, reopened, ready_for_review]
   ```
   The existing job-level `if: draft == false` guard already handles the
   non-draft predicate correctly — no change needed there.

**Rationale:** With AISDLC-218, no PR is ever opened as non-draft (they all go
through `gh pr create --draft`). The `opened` event for `auto-enable-auto-merge`
currently fires but the job is skipped by the draft guard. After AISDLC-218,
`opened` events are draft-opens and the job correctly skips. But `ready_for_review`
events are never captured — so auto-merge is never enabled. Adding
`ready_for_review` to the types list restores auto-merge enablement on the flip.

## Transition behavior (before workflow edits land)

Until the workflow edits above are applied:

- Draft PR opens may still trigger some workflows (those without the draft guard).
- `verify-attestation.yml` will post `failure` on draft open (envelope not at HEAD yet), then `success` after `gh pr ready` (2 CI runs still happen for attestation check).
- `auto-enable-auto-merge.yml` will NOT enable auto-merge (because `ready_for_review` event is not captured) — the operator must enable it manually via `gh pr merge --auto --rebase <number>`.

The AISDLC-218 developer-side changes (draft PR open + `gh pr ready` step) are
safe to ship before the workflow edits — the pipeline is correct, just not at
full efficiency until the workflows are updated.
