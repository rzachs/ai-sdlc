## Final Summary

<!-- SECTION:FINAL_SUMMARY:BEGIN -->
### Summary
Phase 1 of RFC-0035 ships the Decision Catalog substrate: a JSON Schema for the `Decision` resource, an append-only event log at `.ai-sdlc/_decisions/events.jsonl` (event-sourced per OQ-1), and a `cli-decisions` CLI with `list`, `show`, `add` (and `log-path`) subcommands. Everything is gated behind the `AI_SDLC_DECISION_CATALOG=experimental` feature flag; read paths degrade open when the flag is off, the mutating `add` path refuses.

### Changes
- `spec/schemas/decision.v1.schema.json` (new): JSON Schema for the materialized Decision view per §11 sketch + extends to `decisionLog[]` carrying the full append-only event history.
- `pipeline-cli/src/decisions/` (new module): `decision-record.ts` (types + validators), `event-log.ts` (append + read + id allocation + opened-event factory), `projection.ts` (events → Decision left fold), `feature-flag.ts` (`AI_SDLC_DECISION_CATALOG` predicate + disabled message), `index.ts` (barrel).
- `pipeline-cli/src/cli/decisions.ts` (new): yargs router for `list / show / add / log-path`. Interactive readline prompt for `add` when stdin is a TTY; flag-driven path (`--summary --scope --option id:desc`) when not. Async handlers throughout so test stubs propagate via `parseAsync` rejection.
- `pipeline-cli/bin/cli-decisions.mjs` (new): bin shim forwarding to `dist/cli/decisions.js`.
- `pipeline-cli/package.json` (modified): new `cli-decisions` bin + `./decisions` subpath export.
- `pipeline-cli/src/index.ts` (modified): re-exports the decisions module's public surface.
- Tests: `decisions/{decision-record,event-log,projection,feature-flag,schema}.test.ts` (45 unit tests, including Ajv2020 schema-validation + round-trip); `cli/decisions.test.ts` (15 in-process CLI tests covering every AC).
- `reference/src/core/generated-schemas.ts` + `ai-sdlc-plugin/mcp-server/dist/bin.js`: regenerated by `pnpm build` to pick up the new schema (autogenerated, no manual edits).

### Design decisions
- **Event-sourced storage** (OQ-1): the log is the source of truth; `Decision` is a projection. Phase 1 only emits `decision-opened`, but the projection tolerates every OQ-1 event-type for forward-compat (Phase 2+ event types fold into `decisionLog` without crashing a Phase-1 reader).
- **Storage under `.ai-sdlc/_decisions/`** (not `$ARTIFACTS_DIR/`): Decisions are first-class workspace artefacts (the operator wants them checked in / replayed across worktrees), not transient run artefacts. Matches the dor-config convention.
- **Degrade-open on read, refuse on write** (AC#6): a missing flag must not break passive queries, but mutating the log under a disabled flag would poison the calibration corpus when the flag is later enabled — so `add` refuses with exit 1.
- **Async yargs handlers**: documented in `decisions.ts` — yargs only propagates handler errors through `parseAsync` rejection when the handler is async, so every handler that calls `fail()` / `process.exit` is async.
- **Schema validation via Ajv2020**: same pattern as `orchestrator-events.v1.schema.json`. Round-trip test seeds an event, projects the Decision, validates against the published schema.
