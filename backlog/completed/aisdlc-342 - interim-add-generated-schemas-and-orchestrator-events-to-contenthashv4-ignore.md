## Bug

PR #498 (AISDLC-280) was kicked out of the merge queue 3+ times in 2 hours with `ai-sdlc/attestation: failure — contentHashV4 mismatch` on queue probes (`ed47d6bf`, `258d4e2b`, `769e5736`). Root cause: the PR's diff includes 3 high-churn shared files that sibling PRs frequently modify:

1. `reference/src/core/generated-schemas.ts` — auto-regenerated by `pnpm build` whenever ANY new JSON schema lands under `spec/schemas/`
2. `pipeline-cli/src/orchestrator/events.ts` — every RFC adding a new orchestrator event type touches this
3. `spec/schemas/orchestrator-events.v1.schema.json` — same trigger as #2

When the merge queue rebases a PR onto a new main HEAD that includes a sibling PR touching one of these files, the v4 per-file blob SHA changes (the rebase merge produces a new file blob) and v4 invalidates → PR kicked.

The proper fix is contentHashV5 (delta-hash) — see [AISDLC-343](aisdlc-343 - feat-attestation-contenthashv5-rebase-stable-delta-hash.md). This task is the interim fix to reduce the kick frequency while v5 is being designed + shipped (1-day work).

## Acceptance criteria

- [ ] **Add `reference/src/core/generated-schemas.ts` to `CONTENTHASHV4_IGNORE_FILES`** in `orchestrator/src/runtime/attestations.ts`. This file is auto-generated from `spec/schemas/*.schema.json` (the source-of-truth). Reviewer signal is preserved: the underlying schema JSON is still in the hash. Justified exception to the CLAUDE.md "no source files in ignore list" rule because this file carries zero human-authored intent.

- [ ] **Update CLAUDE.md** (the "Review attestations" section) to document the exception with the rationale: "generated-schemas.ts is regenerated from spec/schemas/*.schema.json on every `pnpm build`; including it in v4 would invalidate every PR touching it whenever a sibling PR adds a schema. The hash still covers `spec/schemas/*.schema.json` so the reviewer signal is preserved."

- [ ] **Update `scripts/verify-attestation.mjs`** to mirror the new ignore list (the v4 collector is duplicated on the signer + verifier sides per AISDLC-258 — both must change atomically).

- [ ] **Add a test** in `orchestrator/src/runtime/attestations.test.ts`: sign at SHA X with generated-schemas.ts content "A"; rebase such that the file's blob becomes content "B"; verify against rebased HEAD → assert v4 passes.

- [ ] **DO NOT add** `pipeline-cli/src/orchestrator/events.ts` or `spec/schemas/orchestrator-events.v1.schema.json` to the ignore list. Those carry real review signal — the operator wants to see when an event-payload contract changes. The cost (occasional manual re-sign) is acceptable; only generated files where there's NO human intent in the diff qualify for the exception.
