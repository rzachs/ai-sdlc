## Description

<!-- SECTION:DESCRIPTION:BEGIN -->
Phase 3 of RFC-0009. Four framework resource types gain soul-scoping fields per §8. In-soul variation patterns (variant + journey) are explicitly OUT OF SCOPE — see RFC-0017 (In-Shard Variant Pattern) and RFC-0018 (In-Shard Journey Pattern), both Reserved, pending normative spec.

## Scope (RFC-0009 §10 Phase 3, §8 resource extensions)

- `AgentRole` resource gains `soulScope` field per §8.1: per-role scope to a Soul DID slug or `platform`.
- `AdapterBinding` resource gains `soulScope` field per §8.2: per-binding scope.
- `ProvenanceRecord` resource gains `soulScope` + `tessellatedSoulRef` fields per §8.3: provenance traces per soul.
- `QualityGate` resource gains `soulScope` field per §8.4: per-gate scope.
- All four resources remain backwards-compatible (omitted soulScope = platform-wide, existing behavior).
- In-soul variation patterns NOT shipped here (RFC-0017 + RFC-0018 carve-outs).

## Final Summary

### Summary
Phase 3 of RFC-0009 lands the soul-scoping field surface on `AgentRole`, `AdapterBinding`, `ProvenanceRecord`, and `QualityGate`. Implementation follows the normative §8 spec rather than the literal `soulScope`-everywhere wording of the task ACs — each resource gains the soul-scoping surface RFC-0009 §8.1-§8.4 actually specifies (e.g., `AgentRole.scope` + `soulBindings`; `AdapterBinding.soulOverrides`; `ProvenanceRecord.targetedSouls` + `substrateScoped` + `tessellatedSoulRef`; `QualityGate.soulScope`). All four fields are optional; omission preserves single-DID behavior for backward compatibility.

### Changes
- `spec/schemas/agent-role.schema.json` (modified): adds `scope: 'platform' | 'soul' | 'tenant'` and `soulBindings: string[]` under `spec` (RFC-0009 §8.1).
- `spec/schemas/adapter-binding.schema.json` (modified): adds `soulOverrides: [{ soul, config }]` under `spec` (RFC-0009 §8.2).
- `spec/schemas/quality-gate.schema.json` (modified): adds `soulScope: string` (Soul DID URI) under `spec` (RFC-0009 §8.4).
- `reference/src/core/types.ts` (modified): `AgentRoleScope` type alias; `AgentRoleSpec.scope` + `AgentRoleSpec.soulBindings`; `AdapterSoulOverride` interface; `AdapterBindingSpec.soulOverrides`; `QualityGateSpec.soulScope`.
- `reference/src/core/provenance.ts` (modified): `ProvenanceRecord.targetedSouls` + `.substrateScoped` + `.tessellatedSoulRef`; `createProvenance()` threads the new fields through; `provenanceToAnnotations()` / `provenanceFromAnnotations()` serialize round-trip (JSON-encoded array for `targetedSouls`, `'true'/'false'` for `substrateScoped`, plain string for `tessellatedSoulRef`).
- `reference/src/core/soul-scoping-phase3.test.ts` (new): 31 hermetic tests covering soul-scoped, platform-wide, mixed-scope, and rejection scenarios per resource; one cross-resource three-soul-platform integration test exercising the §11 worked-example shape.
- `reference/src/core/generated-schemas.ts` (regenerated by `prebuild`): inlines the updated `*.schema.json` files.

### Design decisions
- **Followed RFC normative spec, not literal AC wording.** Task ACs use the shorthand "soulScope field" for all four resources; RFC-0009 §8.1-§8.4 specify materially different surfaces per resource (`scope` + `soulBindings` for AgentRole, `soulOverrides[]` for AdapterBinding, `targetedSouls` + `substrateScoped` + `tessellatedSoulRef` for ProvenanceRecord, `soulScope` for QualityGate). The RFC is the normative source; ACs were interpreted as "each resource gains its §8.X soul-scoping surface" and the existing un-committed test file (which appears to have been generated by a prior dispatch attempt) was already aligned with this reading.
- **Soul URIs are typed as `string`, not pattern-validated.** Phase 1's `design-intent-document.schema.json` declares `parentTessellation` as a free-form `string`. Phase 3 mirrors this — soul URIs flow through unconstrained until a framework-wide URI grammar is settled. Validation can tighten later without breaking existing DIDs.
- **Round-trip serialization for ProvenanceRecord uses JSON encoding for `targetedSouls`.** Annotation values are strings (`Record<string, string>`); a JSON-encoded array survives round-trip while a comma-joined list would lose information for URIs containing commas (rare but possible). `substrateScoped` round-trips as `'true'/'false'`; missing / non-`'true'` / non-`'false'` parses as `undefined` to keep legacy records clean.
- **Defensive parsing on the deserializer side.** Malformed `targetedSouls` JSON is treated as absent rather than throwing — preserves the existing "missing required fields → return undefined" contract.
