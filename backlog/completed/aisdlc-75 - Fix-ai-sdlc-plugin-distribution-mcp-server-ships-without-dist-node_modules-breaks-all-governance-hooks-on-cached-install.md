## Description

<!-- SECTION:DESCRIPTION:BEGIN -->
## Context

User report (2026-04-28) on ai-sdlc plugin v0.7.0:

> The cached `ai-sdlc` plugin v0.7.0 mcp-server had no `dist/` folder and no `node_modules/`. Plugin manifest points to `dist/index.js` as the MCP entry point, but the file didn't exist. Claude tried to spawn the MCP server, the binary wasn't there, plugin reconnect failed silently.

User had to manually run `pnpm install && pnpm run build` inside `~/.claude/plugins/cache/ai-sdlc/ai-sdlc/0.7.0/mcp-server/` to make the plugin work.

## Root cause

1. `.gitignore` excludes `dist/` (correct for source repos)
2. Marketplace clones the source tree at tag `v0.7.0` from `./ai-sdlc-plugin/` → no `dist/` arrives
3. Marketplace doesn't run `pnpm install` or `pnpm build` post-fetch (no post-install hook in the plugin protocol)
4. Plugin manifest (`ai-sdlc-plugin/.claude-plugin/plugin.json` + `ai-sdlc-plugin/plugin.json`) references `mcp-server/dist/bin.js`
5. Spawn fails silently → SessionStart hook, PreToolUse, PostToolUse, MCP-exposed tools (`/ai-sdlc:triage`, `/ai-sdlc:status`, `/ai-sdlc:fix-pr`, `/ai-sdlc:detect-patterns`) all return skill text only with no actual orchestrator data

## TypeScript errors that surfaced (compounding factor)

The user got partial JS emitted due to TS bailout, but on a clean build:

- `src/tools/list-detected-patterns.ts:9` — TS2589: type instantiation excessively deep
- `src/tools/list-detected-patterns.ts:60` — TS2802: MapIterator needs `--downlevelIteration` or `target: es2015+`
- `src/tools/list-detected-patterns.ts:75` — same TS2802
- ZodNumber vs AnySchema overload mismatch (MCP SDK version drift — `@modelcontextprotocol/sdk@^1.26.0` vs `zod@^3.23.0`)

## Two angles to fix (do BOTH)

### Angle A — make `dist/` self-contained AND present at install time

Best path: **bundle the mcp-server with esbuild** into a single self-contained `dist/bin.js` (all runtime deps inlined). Then either:

- (a) commit the bundled `dist/bin.js` to the repo (override `.gitignore` for that one path), OR
- (b) build it as part of a release workflow that creates a tag and the marketplace fetches the tag's archive (which includes dist), OR
- (c) publish the mcp-server to npm and the plugin manifest references the npm-installed binary

Recommend (a) — simplest, most predictable, no external publish step. Trade-off: every plugin version commit includes a regenerated bundle (a few hundred KB), which is git-noisy but clear.

### Angle B — fix the TS build so partial JS isn't a coping mechanism

- Refactor `list-detected-patterns.ts` to avoid TS2589 (excessively deep type — usually means a recursive Zod schema or generic that needs explicit typing/breaking)
- Set `tsconfig.json` `target: "es2022"` or add `downlevelIteration: true` so Map iteration works
- Pin or bump `@modelcontextprotocol/sdk` to a version compatible with the `zod` version in use; OR pin `zod` to a version the SDK accepts

## Out of scope

## Impact when broken (motivates priority)

When mcp-server fails to load, ALL of these silently break:
- SessionStart hook — governance context not injected
- PreToolUse — destructive Bash actions (`gh pr merge`, `git push --force`) not blocked
- PostToolUse — telemetry not collected
- MCP tools — `/ai-sdlc:status`, `/ai-sdlc:triage`, `/ai-sdlc:fix-pr`, `/ai-sdlc:detect-patterns` return skill prompt text instead of orchestrator data
- AISDLC-71's `/ai-sdlc execute` flow may proceed without governance context (the SubagentStart hook from AISDLC-71 is separate so it survives, but the ambient governance does not)

This is silent: the user doesn't see an error. They just notice "tools aren't doing what they should" — exactly the situation the user reported.

## Final Summary

## Summary

Fixed AISDLC-75 by bundling the ai-sdlc-plugin MCP server with esbuild into a single self-contained `dist/bin.js`, un-ignoring that one file in `.gitignore` so it ships in marketplace clones (which never run `pnpm install`), bumping `zod` to `^3.25.76` to satisfy the SDK peer requirement, and adding a `Verify MCP Bundle` GitHub workflow that gates four invariants on every PR (presence, valid ESM, self-containment via live MCP initialize handshake with no `node_modules`, and freshness vs. a clean rebuild).

## Changes

- `.gitignore` (modified): un-ignore `ai-sdlc-plugin/mcp-server/dist/bin.js` only (other dist files stay ignored)
- `pnpm-lock.yaml` (modified): zod bumped to ^3.25.76
- `ai-sdlc-plugin/mcp-server/package.json` (modified): zod range bumped, esbuild devDep added, `bundle` script added, `build` script chains `tsc && pnpm bundle`
- `ai-sdlc-plugin/mcp-server/CHANGELOG.md` (new): documents the fix
- `ai-sdlc-plugin/mcp-server/scripts/bundle.mjs` (new): esbuild config — ESM target node20, `packages: 'bundle'` to inline all deps, `external: ['node:*']`, createRequire banner for CJS interop, shebang
- `ai-sdlc-plugin/mcp-server/scripts/verify-bundle.mjs` (new): four-invariant verifier (presence + valid ESM + self-contained via quarantine+spawn+initialize + freshness via sha256 byte-exact rebuild diff)
- `ai-sdlc-plugin/mcp-server/src/bundle.test.ts` (new): vitest smoke test exercising the same MCP initialize handshake from a tmpdir cwd with cleared NODE_PATH
- `ai-sdlc-plugin/mcp-server/dist/bin.js` (new, committed bundle): 734KB self-contained ESM with @modelcontextprotocol/sdk + zod inlined
- `.github/workflows/verify-mcp-bundle.yml` (new): runs the verifier on every PR touching `ai-sdlc-plugin/**` plus the workflow itself

## Design decisions

- **Path-(a) commit the bundle**, not path-(b) release-workflow that builds at tag time. Reasons: simplest, no external publish step, every change to the bundle is reviewable as a PR diff (large but visible). Cost: each plugin-version commit includes a regenerated bundle; CI's freshness check via sha256 + frozen lockfile prevents hand-edited bundles.
- **Smoke test does NOT quarantine node_modules** in the in-process vitest (deliberately; race-prone with parallel suites). It uses tmpdir cwd + NODE_PATH='' instead. The CI gate (`verify-bundle.mjs`) IS the authoritative quarantine-then-initialize check — single-process, safe to move.
- **MCP initialize handshake over stdio** is the strongest possible self-containment signal: it proves the bundle loads, the SDK is inlined, the server starts, and stdio works.
- **TS errors mentioned in the original task description weren't actually present at HEAD** — the dev confirmed `pnpm --filter @ai-sdlc/plugin-mcp-server build` was already clean before any changes. Most likely fixed by an unrelated commit between when the task was filed and now. The dev didn't fabricate fixes.

## Follow-up

- **AC #5 (operator end-to-end install verification)** deferred per task design — cannot be exercised from inside the dev subagent. Operator runs: `rm -rf ~/.claude/plugins/cache/ai-sdlc-framework/ai-sdlc/v0.7.0` + restart Claude Code + confirm `/ai-sdlc:status` returns real data instead of skill text. Cite the verification result in a comment on the merged PR or in a follow-up task.
- **Pin esbuild to exact `0.27.3`** (not `^0.27.3`) — the freshness check is byte-exact sha256; any 0.27.x patch that tweaks output would break CI on PRs that didn't touch the bundle source. Frozen-lockfile in CI mitigates this for now but a one-line pin would close the gap. Filed as small follow-up.
- **`renameSync` cross-FS portability**: verifier moves `node_modules` to `/tmp/`; on systems where `/tmp` is a separate filesystem (tmpfs), this throws EXDEV. Standard GHA ubuntu colocates them, so works in CI today; consider falling back to `dist-quarantine-<rand>` inside the package root for portability.
- **SIGTERM handler in verify-bundle.mjs**: `restore()` runs on `'exit'` and `'SIGINT'` but not `'SIGTERM'` or uncaughtException. Local devs whose run gets killed could be left with quarantined node_modules. Low blast radius (CI runners are ephemeral) but worth a SIGTERM handler.
- **Missing workflow `permissions:` block** in `verify-mcp-bundle.yml` — inherits repo default. Add explicit `permissions: contents: read` at workflow or job level for principle-of-least-privilege.
- **Reviewer-suggestion duplication** between `bundle.test.ts` and `verify-bundle.mjs` (MCP initialize handshake logic) — small helper would DRY, but acceptable per project's "three similar lines > premature abstraction".
