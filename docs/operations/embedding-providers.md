## Configuration overview

The embedding framework is **off by default** in v1. Enable it by setting:

```bash
AI_SDLC_EMBEDDING_PROVIDER=1
```

When enabled, `Pipeline.spec.embedding` controls the provider and storage:

```yaml
spec:
  embedding:
    provider: openai-text-embedding-3-small   # adapter name
    storageBackend: jsonl                      # default; only option in v1
    staleVectorPolicy: lazy-re-embed           # 'lazy-re-embed' | 'fail-loud' | 'warn'
    gcRetentionDays: 90                        # per-org GC threshold
```

Per-org defaults live in `.ai-sdlc/embedding-config.yaml` (generated by `ai-sdlc init`).

### Required environment variables

| Adapter | Required env var |
|---|---|
| `openai-text-embedding-3-small` | `OPENAI_API_KEY` |
| Future adapters | Declared in adapter's `requires.envVar` |

## Running embedding GC

The `cli-embedding-gc` tool removes stale vector store entries based on their
`writtenAt` timestamp. Run it periodically (e.g., weekly via cron) or manually
after large batch operations.

### Basic usage

```bash
# Run GC with default 90-day retention.
node pipeline-cli/bin/cli-embedding-gc.mjs run \
  --artifacts-dir .ai-sdlc/artifacts

# Per-org retention override (30 days):
node pipeline-cli/bin/cli-embedding-gc.mjs run \
  --artifacts-dir .ai-sdlc/artifacts \
  --retention-days 30

# Restrict to one provider:
node pipeline-cli/bin/cli-embedding-gc.mjs run \
  --artifacts-dir .ai-sdlc/artifacts \
  --provider openai-text-embedding-3-small
```

### Dry-run (preview without modifying files)

```bash
node pipeline-cli/bin/cli-embedding-gc.mjs run \
  --artifacts-dir .ai-sdlc/artifacts \
  --dry-run
# Output: [cli-embedding-gc] dry-run: would remove 142 of 1024 entries (older than 90 days)
```

### JSON output (for scripting)

```bash
node pipeline-cli/bin/cli-embedding-gc.mjs run \
  --artifacts-dir .ai-sdlc/artifacts \
  --format json
# Output: {"removed":142,"scanned":1024,"filesProcessed":2,"retentionDays":90}
```

### Retention boundary semantics

- Entries with `writtenAt` **older than** `retentionDays` days are removed.
- Entries written **exactly at** the boundary are retained (inclusive lower bound).
- Entries **without a `writtenAt` field** are treated as legacy/corrupt and retained
  (not silently dropped).

## Inspecting the vector store

The `stats` subcommand provides a read-only view of the current store:

```bash
node pipeline-cli/bin/cli-embedding-gc.mjs stats \
  --artifacts-dir .ai-sdlc/artifacts

# Table output:
Provider                          ModelVersion  Count  OldestEntry  NewestEntry
--------------------------------  ------------  -----  -----------  -----------
openai-text-embedding-3-small     2024-01-25    1024   2026-01-01   2026-05-23
```

```bash
# JSON output for scripting:
node pipeline-cli/bin/cli-embedding-gc.mjs stats \
  --artifacts-dir .ai-sdlc/artifacts \
  --format json
```

The underlying JSONL files are human-readable and `jq`-friendly:

```bash
# Count entries by provider:
jq -r '.embeddingProvider' .ai-sdlc/artifacts/_embeddings/*.jsonl | sort | uniq -c

# Find entries written in the last 7 days:
jq 'select(.writtenAt > "2026-05-16")' .ai-sdlc/artifacts/_embeddings/*.jsonl

# Verify an entry's textHash:
jq 'select(.textHash == "<hash>")' .ai-sdlc/artifacts/_embeddings/*.jsonl
```

## Scale escalation

The JSONL backend is designed for corpora up to **~100K vectors per provider+version**.
When you cross either threshold:

- **> 100,000 entries** per `(provider, modelVersion)` tuple
- **p95 read latency > 250ms** for `read()` by textHash

...the backend emits an operator-visible signal (logged to stderr or your telemetry
callback) pointing to this runbook section.

### When to escalate

| Signal | Recommended action |
|---|---|
| Count > 100K | Evaluate sqlite backend (Phase 6+) or archive old entries via GC |
| p95 read > 250ms | Run GC to reduce corpus size; monitor; escalate to sqlite if persistent |
| Both thresholds crossed | Swap to sqlite backend immediately |

### Escalation path: JSONL → sqlite

> **Note:** sqlite backend is scheduled for Phase 6+ and is not yet available in v1.
> When it ships, the swap procedure will be:

1. Run a full GC pass to remove stale entries: `cli-embedding-gc run --retention-days 90`
2. Update `Pipeline.spec.embedding.storageBackend` to `'sqlite'`
3. Run `cli-embedding-bump --migrate-storage` to rewrite the JSONL files to sqlite
4. Verify the migration: `cli-embedding-gc stats` should show 0 JSONL entries
5. Monitor p95 read latency — sqlite indexed lookups should be <1ms for 1M entries

The framework's `EmbeddingStorageBackend` interface (at
`orchestrator/src/embedding/storage/types.ts`) is the extension point.
Adopter-supplied sqlite or pgvector backends implement the interface and are
wired into the factory at `orchestrator/src/embedding/storage/index.ts`.

### Escalation criteria (codified)

Per RFC-0019 OQ-1 re-walkthrough (2026-05-21):

> **Scale-escalation heuristic:** swap from JSONL to sqlite (or vector DB) when:
> - Count per `(provider, modelVersion)` exceeds ~100K entries, OR
> - p95 read latency exceeds 250ms.
> 
> Makes the transition operator-visible and corpus-driven, not tribal knowledge.

These thresholds are exported as:
- `SCALE_ESCALATION_MAX_ENTRIES = 100_000`
- `SCALE_ESCALATION_P95_READ_MS = 250`

from `orchestrator/src/embedding/storage/jsonl-backend.ts`.

## Storage layout reference

```
<artifactsDir>/_embeddings/
├── openai-text-embedding-3-small-2024-01-25.jsonl  ← one file per (provider, modelVersion)
└── openai-text-embedding-3-large-2024-01-25.jsonl  ← (if multi-provider in use)
```

### Filename convention

Each file is named `<safeProvider>-<safeModelVersion>.jsonl` where each component is
sanitized to the regex `[a-zA-Z0-9._-]` (unsafe characters replaced with `-`). The
directory listing itself serves as the index — `scan()` walks `<embeddingsDir>/*.jsonl`
and each entry carries its own `(embeddingProvider, embeddingModelVersion)` provenance
for filtering. There is no separate index file (iter-2 review: dropping `_index.json`
eliminates the read-modify-write race on concurrent first-writes for different
provider/version tuples).

### JSONL entry format

Each line in a `.jsonl` file is a JSON object (the `VectorStoreEntry` schema,
`spec/schemas/vector-store-entry.v1.schema.json`):

```json
{
  "vector": [0.1, 0.2, 0.3, ...],
  "embeddingProvider": "openai-text-embedding-3-small",
  "embeddingModelVersion": "2024-01-25",
  "writtenAt": "2026-05-23T12:00:00.000Z",
  "text": "original source text",
  "textHash": "sha256hexhash",
  "metadata": { "sourceDoc": "rfc-0009.md" }
}
```

**Why `text` is stored:** Migration (`cli-embedding-bump`) requires re-embedding the
original text when a deprecated adapter is retired. Storing the vector without the
text makes migration impossible.

**Why `textHash` is stored:** Read-side deduplication uses the hash as the cache key
without recomputing it on every lookup.

**Provenance fields (`embeddingProvider`, `embeddingModelVersion`):** Both fields are
part of the vector's identity per RFC-0019 §2.3. Cross-provider comparisons are
mathematically invalid (different embedding spaces). The storage layout enforces
isolation by keeping one file per `(provider, modelVersion)` tuple.
